## ADDED Requirements

### Requirement: Multi-model ChatClient registration

The system SHALL register separate `ChatClient` beans for each configured model provider (DeepSeek, MiMo, DashScope), each backed by its own `ChatModel` implementation.

#### Scenario: DeepSeek ChatClient available
- **WHEN** application starts with DeepSeek API key configured
- **THEN** a `ChatClient` bean named `deepseekChatClient` is registered and functional

#### Scenario: DashScope ChatClient available
- **WHEN** application starts with DashScope API key configured
- **THEN** a `ChatClient` bean named `dashscopeChatClient` is registered and functional

#### Scenario: MiMo ChatClient available
- **WHEN** application starts with MiMo API key configured (via OpenAI-compatible endpoint)
- **THEN** a `ChatClient` bean named `mimoChatClient` is registered and functional

### Requirement: Dynamic model switching via API parameter

The system SHALL expose a REST endpoint `POST /api/chat` that accepts a `model` query parameter to select which model provider handles the request.

#### Scenario: Switch to DeepSeek
- **WHEN** client sends `POST /api/chat?model=deepseek` with `{"message": "Hello"}`
- **THEN** the request is routed to the DeepSeek ChatClient and returns a response generated by DeepSeek

#### Scenario: Switch to DashScope
- **WHEN** client sends `POST /api/chat?model=dashscope` with `{"message": "Hello"}`
- **THEN** the request is routed to the DashScope ChatClient and returns a response generated by Qwen

#### Scenario: Invalid model name
- **WHEN** client sends `POST /api/chat?model=unknown` with `{"message": "Hello"}`
- **THEN** the system returns HTTP 400 with an error message listing valid model names

### Requirement: Streaming chat response support

The system SHALL support Server-Sent Events (SSE) streaming for chat responses via `GET /api/chat/stream`.

#### Scenario: Stream chat response
- **WHEN** client sends `GET /api/chat/stream?model=deepseek&message=Hello`
- **THEN** the response is streamed as SSE events with `Content-Type: text/event-stream`

### Requirement: Multi-model configuration via application.yml

The system SHALL read model API keys, base URLs, and model names from `application.yml` under the `app.models` namespace.

#### Scenario: All models configured
- **WHEN** `application.yml` contains valid configuration for all three models
- **THEN** all three ChatClient beans are registered at startup

#### Scenario: Partial model configuration
- **WHEN** only DeepSeek and DashScope API keys are configured (MiMo key is empty)
- **THEN** DeepSeek and DashScope ChatClients are registered; MiMo ChatClient is skipped gracefully
