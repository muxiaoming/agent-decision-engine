# Feature Specification: Tool 注解重构与代码规范化

**Feature Branch**: `002-tool-refactor-code-style`

**Created**: 2026-06-04

**Status**: 草稿

**Input**: 用户描述："将代码改为 @Tool 注解形式，使用 ToolCallback 统一管理所有工具，ChatClient 调用时不硬编码工具名；代码风格严格遵循企业 Java 标准，禁止使用 var，必须显式声明类型，代码要干净、规范、可直接用于生产环境"

## 用户场景与测试 *（必填）*

### 用户场景 1 — 工具定义方式重构（优先级: P1）

作为开发者，我希望项目中的 Function Calling 工具从当前的 `Function<T, R>` Bean 注册方式
改为 Spring AI 的 `@Tool` 注解方式定义，使工具代码更直观、更易维护，且工具的名称和
描述直接由注解属性声明，无需额外的 `@Description` 注解。

**优先级理由**: 工具定义方式是 Function Calling 模块的核心，改变定义方式将影响工具注册、
工具发现和工具调用的全链路。这是本次重构的首要任务。

**独立测试**: 定义一个带 `@Tool` 注解的工具类，验证 Spring AI 能正确识别该工具并注册
为可调用工具，工具名称和描述与注解属性一致。

**验收场景**:

1. **假设** 开发者定义了一个带 `@Tool(description="查询天气")` 注解的天气查询方法,
   **当** 应用启动时,
   **则** 系统自动将该方法注册为可调用工具，工具描述为"查询天气"。
2. **假设** 工具方法有明确的参数类型定义,
   **当** AI 模型需要调用该工具时,
   **则** 系统能正确解析参数并调用工具方法。
3. **假设** 工具类中的方法未加 `@Tool` 注解,
   **当** 应用启动时,
   **则** 该方法不会被注册为可调用工具。

---

### 用户场景 2 — ToolCallback 统一管理（优先级: P1）

作为开发者，我希望所有工具通过 `ToolCallback` 机制统一注册和管理，ChatClient 调用
时通过注入的 `ToolCallback[]` 数组或 `ToolCallbackProvider` 传递可用工具，而不是
在代码中硬编码工具名称字符串（如 `toolNames("weatherFunction", "calculatorFunction")`）。

**优先级理由**: 硬编码工具名称违反了开闭原则，新增工具需要同时修改多处代码。统一管理
消除了这种耦合，使新增工具只需添加类和注解即可生效。

**独立测试**: 新增一个工具类（如"翻译工具"），无需修改 ChatClient 调用代码，验证
该工具自动对 ChatClient 可用。

**验收场景**:

1. **假设** 系统已通过 ToolCallback 注册了多个工具,
   **当** ChatClient 发起带工具调用的对话时,
   **则** 所有已注册工具自动可用，无需在 ChatClient 调用处硬编码工具名。
2. **假设** 开发者新增了一个工具类并添加 `@Tool` 注解,
   **当** 应用重启后,
   **则** 新工具自动对 ChatClient 可用，无需修改任何已有调用代码。
3. **假设** 需要在特定场景下只使用部分工具,
   **当** 开发者通过配置指定工具子集时,
   **则** 只有指定的工具被传递给 ChatClient。

### 用户场景 3 — 企业级代码风格规范化（优先级: P2）

作为技术负责人，我希望项目中所有 Java 代码严格遵循企业级编码标准：禁止使用 `var`
关键字（必须显式声明所有变量类型），代码结构清晰、命名规范、注释完整，达到可直接
用于生产环境的质量水平。

**优先级理由**: 代码风格规范化虽然不改变功能，但直接影响代码的可读性、可维护性和
团队协作效率。在演示项目中展示企业级代码标准具有示范意义。

**独立测试**: 使用静态分析工具或人工审查验证代码中无 `var` 关键字，所有变量、参数、
返回值均有显式类型声明。

**验收场景**:

1. **假设** 审查任意一个 Java 源文件,
   **当** 检查变量声明时,
   **则** 所有局部变量均使用显式类型声明，不存在 `var` 关键字。
2. **假设** 审查任意 Service 或 Controller 类,
   **当** 检查代码结构时,
   **则** 类职责单一、方法长度合理（不超过 30 行）、命名语义清晰。
3. **假设** 审查所有 record 类型的 DTO,
   **当** 检查字段命名和类型时,
   **则** 字段名语义明确，类型正确使用包装类或基本类型。

---

### 用户场景 4 — 全链路一致性验证（优先级: P2）

作为开发者，我希望重构后的代码在功能上与重构前完全一致：天气查询、数学计算等
工具调用行为不变，所有现有测试继续通过，不引入回归问题。

**优先级理由**: 重构的核心原则是行为不变。必须确保重构后所有功能和测试与重构前
等价。

**独立测试**: 运行 `mvn test` 全部通过，并通过手动验证 Function Calling 端点的
工具调用行为与重构前一致。

**验收场景**:

1. **假设** 重构完成后运行 `mvn test`,
   **当** 所有测试执行时,
   **则** 全部通过，无失败或错误。
2. **假设** 通过发送"北京今天天气如何"到工具调用端点,
   **当** AI 模型处理请求时,
   **则** 自动调用天气工具并返回与重构前一致格式的回答。
3. **假设** 发送普通闲聊问题到工具调用端点,
   **当** AI 模型处理请求时,
   **则** 不触发任何工具调用，直接返回回答。

### 边界场景

- 当 `@Tool` 注解的方法参数类型不被 Spring AI 支持时，系统如何处理？
  应在启动时报错，提供明确的类型不支持提示。
- 当多个工具类中存在同名方法时，系统如何处理？
  应通过工具名称唯一性校验，启动时检测冲突并报错。
- 当 ToolCallback 数组为空（无任何工具注册）时，ChatClient 调用行为如何？
  应正常工作，不触发工具调用，与无工具模式一致。
- 当 `var` 关键字被意外引入时，如何预防？
  应通过 Checkstyle 或编译器配置禁止 `var` 使用。

## 需求 *（必填）*

### 功能需求

- **FR-001**: 所有 Function Calling 工具必须使用 `@Tool` 注解定义，包括天气查询工具和
  数学计算工具。
- **FR-002**: 工具的描述信息（供模型理解）必须通过 `@Tool` 注解的 `description` 属性
  声明，不得依赖额外注解。
- **FR-003**: 所有工具必须通过 `ToolCallback` 机制统一注册，ChatClient 调用时通过注入
  `ToolCallback[]` 或等效机制传递可用工具，禁止在代码中硬编码工具名称字符串。
- **FR-004**: 新增工具只需创建带 `@Tool` 注解的类即可自动注册，无需修改 ChatClient
  调用代码或其他配置文件。
- **FR-005**: 所有 Java 源文件中禁止使用 `var` 关键字，所有变量、参数、返回值必须使用
  显式类型声明。
- **FR-006**: 代码必须遵循企业级 Java 编码规范：类职责单一、方法长度不超过 30 行、
  命名语义清晰、适当的 JavaDoc 注释。
- **FR-007**: 重构后的 Function Calling 功能必须与重构前行为完全一致，所有现有测试
  继续通过。
- **FR-008**: 代码中不得出现硬编码的工具名称字符串引用，工具发现和注册必须是自动的。

### 关键实体

- **Tool（工具）**: 代表一个可被 AI 模型调用的能力单元，包含名称、描述、输入参数
  定义、执行逻辑。通过 `@Tool` 注解声明。
- **ToolCallback（工具回调）**: 代表 Spring AI 框架中工具的运行时表示，封装了工具的
  调用协议。系统中所有工具统一管理为 ToolCallback 集合。
- **ChatClient（对话客户端）**: 代表与 AI 模型交互的客户端，调用时自动携带已注册的
  ToolCallback 集合。

## 成功标准 *（必填）*

### 可衡量结果

- **SC-001**: 所有 Function Calling 工具均通过 `@Tool` 注解定义，代码中不存在旧的
  `Function<T, R>` Bean 注册方式。
- **SC-002**: ChatClient 调用代码中不存在任何硬编码的工具名称字符串，工具注册完全
  自动化。
- **SC-003**: 新增工具只需一个带注解的类文件，无需修改任何已有代码，新增工具后
  测试全部通过。
- **SC-004**: 所有 Java 源文件中 `var` 关键字使用数量为零。
- **SC-005**: 重构后全部测试通过，无回归问题。
- **SC-006**: 代码可直接用于生产环境：无 TODO 残留、无调试代码、注释完整、命名规范。

## 假设

- 项目使用 Spring AI 1.0.x 版本，该版本已支持 `@Tool` 注解和 `ToolCallback` 机制。
- 重构范围仅限 Function Calling 模块（tools 包）和 ChatClient 调用相关代码，不涉及
  其他模块（RAG、Graph、流式输出等）的架构变更。
- 代码风格规范化仅针对本次重构涉及的文件，不要求一次性修改项目中所有已有文件。
  但重构过程中接触到的文件必须遵循新标准。
- `var` 关键字禁令仅适用于业务源代码（src/main/java 和 src/test/java），不适用于
  配置文件或构建脚本。
