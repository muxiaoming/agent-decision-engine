# 功能规格说明：Spring AI + Langfuse3 完整演示项目

**功能分支**: `001-springai-langfuse3-demo`

**创建日期**: 2026-06-03

**状态**: 草稿

**输入**: 用户描述："开发一个完整的 Spring AI 演示项目，包含多模型切换、三种流式输出、RAG、Function Calling、Graph 工作流、Langfuse3 可观测平台集成"

## 用户场景与测试 *（必填）*

### 用户场景 1 — 多模型自由切换对话（优先级: P1）

作为开发者，我希望在同一个演示项目中同时接入 DeepSeek、字节 MiMo、
阿里通义千问三个 AI 模型，通过简单的配置或接口参数即可在三者之间
自由切换，以便对比不同模型的回答质量和性能表现。

**优先级理由**: 多模型切换是整个演示项目的基础能力，所有后续功能
（流式输出、RAG、Function Calling）都依赖于此。没有模型接入，
其他功能无从谈起。

**独立测试**: 向三个模型分别发送同一问题，验证每个模型均能正常返回
回答，且切换模型时无需重启应用。

**验收场景**:

1. **假设** 应用已启动且三个模型的 API 密钥已配置,
   **当** 用户选择 DeepSeek 模型并发送问题,
   **则** 系统使用 DeepSeek 模型返回回答。
2. **假设** 用户正在使用 DeepSeek 模型对话,
   **当** 用户切换到 MiMo 模型并发送同一问题,
   **则** 系统使用 MiMo 模型返回回答，且两次回答来自不同模型。
3. **假设** 用户请求使用一个未配置 API 密钥的模型,
   **当** 用户发送对话请求,
   **则** 系统返回明确的错误提示，说明该模型不可用。

---

### 用户场景 2 — 三种流式实时对话（优先级: P1）

作为开发者，我希望体验 AI 对话的实时流式输出效果，项目应提供三种
主流流式方案（SSE 标准流式、SseEmitter 流式、Flux 响应式流式），
每种方案有独立的接口入口，以便我了解不同技术方案的实现差异和
适用场景。

**优先级理由**: 流式输出是 LLM 应用的核心用户体验。演示三种方案
可为不同技术栈的开发者提供参考，是本项目的第二大核心价值。

**独立测试**: 分别调用三个流式接口，验证每个接口都能以流式方式
逐步返回 AI 回答内容，而非一次性返回完整结果。

**验收场景**:

1. **假设** 用户通过 SSE 流式接口发送问题,
   **当** AI 模型生成回答时,
   **则** 客户端逐步接收到回答片段，形成打字机效果。
2. **假设** 用户通过 SseEmitter 流式接口发送问题,
   **当** AI 模型生成回答时,
   **则** 客户端以 SSE 事件流的形式逐步接收到回答片段。
3. **假设** 用户通过 Flux 响应式接口发送问题,
   **当** AI 模型生成回答时,
   **则** 客户端以响应式流的方式逐步接收到回答片段。
4. **假设** 流式传输过程中模型服务异常中断,
   **当** 连接断开时,
   **则** 客户端收到错误事件或连接关闭信号，不会无限等待。

---

### 用户场景 3 — RAG 本地知识库问答（优先级: P2）

作为开发者，我希望基于本地文档构建知识库，当用户提问时系统能够
从知识库中检索相关内容并结合大模型生成准确回答，且支持流式输出
以获得良好的交互体验。

**优先级理由**: RAG 是企业级 AI 应用最常见的场景之一，演示文档摄入、
向量嵌入、检索和生成的完整流程对开发者具有很高的参考价值。

**独立测试**: 导入一份测试文档后，针对文档内容提问，验证系统能够
基于文档内容返回准确回答。

**验收场景**:

1. **假设** 系统已导入一份产品说明书文档,
   **当** 用户提问"产品的核心功能有哪些",
   **则** 系统从知识库中检索相关内容，结合大模型生成包含文档
   信息的回答。
2. **假设** 用户提问的内容与知识库无关,
   **当** 系统在知识库中未找到相关内容时,
   **则** 系统基于模型自身知识回答，并明确告知该回答未参考知识库。
3. **假设** 用户通过 RAG 流式接口提问,
   **当** 系统生成回答时,
   **则** 回答以流式方式逐步返回。

---

### 用户场景 4 — Function Calling 工具调用（优先级: P2）

作为开发者，我希望演示 AI 模型根据用户意图自动调用外部工具或函数
的能力，例如查询天气、计算数学表达式等，以了解 Function Calling
的完整工作流程。

**优先级理由**: Function Calling 是 AI 从"聊天机器人"进化为
"智能代理"的关键能力，是进阶开发者必须掌握的核心功能。

**独立测试**: 发送一个需要调用工具的问题（如"北京今天天气如何"），
验证系统能够自动识别意图、调用对应工具、并将结果整合到回答中。

**验收场景**:

1. **假设** 系统已注册天气查询工具,
   **当** 用户提问"北京今天天气如何",
   **则** 系统自动识别需要调用天气工具，调用后将结果整合为
   自然语言回答。
2. **假设** 系统已注册数学计算工具,
   **当** 用户提问"计算 (3+5)*12 的结果",
   **则** 系统调用计算工具并返回准确结果。
3. **假设** 用户的问题不需要调用任何工具,
   **当** 用户发送普通闲聊问题,
   **则** 系统直接由模型回答，不触发工具调用。

---

### 用户场景 5 — Graph 多步骤工作流（优先级: P3）

作为开发者，我希望演示基于图的多步骤 AI 工作流编排能力，包括
多个处理节点和条件分支，以了解如何构建复杂的 AI 处理流程。

**优先级理由**: Graph 工作流代表了 AI 应用编排的高阶形态，适合
在基础功能完成后作为进阶演示。

**独立测试**: 定义一个包含至少两个节点和一个条件分支的图工作流，
输入测试数据后验证各节点按预期顺序执行，条件分支按规则路由。

**验收场景**:

1. **假设** 系统定义了一个内容处理图工作流（分类→处理→输出）,
   **当** 用户提交一段文本,
   **则** 系统按图定义的节点顺序依次处理，最终输出结果。
2. **假设** 图中存在条件分支（如根据内容分类选择不同处理路径）,
   **当** 输入满足某分支条件时,
   **则** 系统沿正确的分支路径执行，而非走默认路径。
3. **假设** 图中某个节点处理失败,
   **当** 异常发生时,
   **则** 系统返回明确的错误信息，指明失败发生在哪个节点。

---

### 用户场景 6 — Langfuse 全链路可观测（优先级: P3）

作为运维人员或开发者，我希望每一次 AI 请求都能在 Langfuse 平台中
查看完整的调用链路，包括请求内容、模型响应、Token 用量、耗时等
关键指标，且支持云端和本地 Docker 两种部署方式。

**优先级理由**: 可观测性是生产级 AI 应用的必备能力，但可以在核心
功能稳定后再集成。

**独立测试**: 发送一个 AI 请求后，在 Langfuse 界面中找到对应的
Trace 记录，验证其中包含完整的调用链路信息。

**验收场景**:

1. **假设** 系统配置了 Langfuse 云端模式,
   **当** 用户发起一次 AI 对话请求,
   **则** 30 秒内在 Langfuse 云端界面可查看到该请求的完整 Trace。
2. **假设** 系统切换到 Langfuse 本地 Docker 模式,
   **当** 用户发起一次 AI 对话请求,
   **则** 30 秒内在本地 Langfuse 界面可查看到该请求的完整 Trace，
   且 Trace 数据与云端模式等效。
3. **假设** 一个 Trace 记录被查看,
   **当** 运维人员打开该 Trace 详情时,
   **则** 可以看到提示词、补全内容、模型名称、Token 用量和耗时。

---

### 边界场景

- 当某个模型的 API 服务不可用时，系统如何处理？应返回明确的错误
  提示而非崩溃或无限等待。
- 当流式传输过程中客户端断开连接时，系统如何处理？应正确清理
  资源，避免内存泄漏。
- 当知识库为空时用户提问，系统如何处理？应基于模型自身知识
  回答并提示知识库无相关内容。
- 当 Function Calling 的目标工具执行超时时，系统如何处理？
  应返回超时错误而非永久等待。
- 当 Langfuse 服务不可达时，AI 请求本身是否受影响？AI 功能
  不应因可观测性服务故障而中断。

## 需求 *（必填）*

### 功能需求

- **FR-001**: 系统必须支持 DeepSeek、MiMo、通义千问三个 AI 模型，
  用户可通过配置或请求参数自由切换。
- **FR-002**: 系统必须提供 SSE 标准流式聊天接口，以 Server-Sent Events
  协议实时推送 AI 回答片段。
- **FR-003**: 系统必须提供 SseEmitter 流式接口，基于 Spring MVC 的
  SseEmitter 实现流式输出。
- **FR-004**: 系统必须提供 Flux 响应式流式接口，基于 Project Reactor
  的 Flux 实现流式输出。
- **FR-005**: 系统必须支持 RAG 本地知识库问答，包括文档摄入、向量嵌入、
  相关性检索和基于检索结果的流式回答生成。
- **FR-006**: 系统必须支持 Function Calling，允许 AI 模型根据用户意图
  自动调用预注册的外部工具，并将工具结果整合到回答中。
- **FR-007**: 系统必须支持基于图的多步骤 AI 工作流编排，包含至少一个
  多节点图和条件分支逻辑。
- **FR-008**: 系统必须集成 Langfuse3 可观测平台，每次 AI 请求的完整
  调用链路（提示词、响应、模型、Token、耗时）均上报至 Langfuse。
- **FR-009**: 系统必须支持 Langfuse 云端和本地 Docker 两种部署模式，
  通过配置属性切换，两种模式产生等效的追踪数据。
- **FR-010**: 系统必须通过 OpenTelemetry 实现全链路追踪，所有涉及模型
  调用的请求均被追踪，无一例外。
- **FR-011**: 系统必须在 Langfuse 服务不可达时保证 AI 核心功能正常运行，
  可观测性故障不得影响业务。
- **FR-012**: 每个流式接口必须有独立的 API 端点，不得共用同一入口。

### 关键实体

- **AI 模型配置**: 代表一个 AI 模型提供者的连接信息，包含模型名称、
  API 端点、API 密钥引用、模型标识符。系统中存在三个实例。
- **对话请求**: 代表一次用户发起的 AI 对话，包含用户消息、所选模型、
  对话标识、时间戳。
- **对话响应**: 代表 AI 模型的回答，包含回答内容、模型名称、Token
  用量、耗时、是否流式。
- **知识库文档**: 代表一份被摄入知识库的本地文档，包含文档路径、
  文档内容、向量嵌入状态。
- **工具定义**: 代表一个可被 Function Calling 调用的外部工具，包含
  工具名称、功能描述、输入参数定义、执行逻辑引用。
- **Trace 记录**: 代表一次请求在 Langfuse 中的完整追踪链路，包含
  请求信息、各环节耗时、Token 指标。

## 成功标准 *（必填）*

### 可衡量结果

- **SC-001**: 用户可以在不修改代码的情况下，通过配置切换三个 AI 模型
  中的任意一个，切换时间不超过 1 秒（应用无需重启）。
- **SC-002**: 三种流式接口均能以打字机效果实时展示 AI 回答，首个
  回答片段在用户发送问题后 3 秒内开始到达。
- **SC-003**: RAG 问答的回答准确率不低于 80%（基于预设知识库内容的
  标准测试问题集）。
- **SC-004**: Function Calling 能够正确识别用户意图并调用对应工具，
  工具调用成功率不低于 90%。
- **SC-005**: Graph 工作流能够按定义的节点顺序和条件分支正确执行，
  包含至少 3 个节点和 1 个条件分支。
- **SC-006**: 所有 AI 请求在完成后 30 秒内可在 Langfuse 界面中查看到
  完整的 Trace 记录。
- **SC-007**: 系统在 Langfuse 服务不可用时仍能正常响应 AI 请求，
  响应时间不受可观测性故障影响。
- **SC-008**: 每个功能模块（模型切换、三种流式、RAG、Function Calling、
  Graph、可观测性）均可独立启动和测试。

## 假设

- 开发者拥有至少一个 AI 模型提供者（DeepSeek/MiMo/通义千问）的
  有效 API 密钥。系统不要求同时配置三个模型的密钥，未配置密钥的
  模型将被标记为不可用。
- 本地开发环境已安装 Java 21、Maven 3.9+ 和 Docker（用于 Langfuse
  自部署模式）。云端 Langfuse 模式不需要 Docker。
- RAG 功能的向量存储使用嵌入式方案（如 SimpleVectorStore），无需
  额外部署外部向量数据库。
- Langfuse 云端版本使用免费套餐即可满足演示需求，无需企业版功能。
- 本项目为演示/学习用途，不要求生产级别的高可用性、水平扩展或多租户
  隔离。
- 项目文档、任务描述、代码注释和 Git 提交备注统一使用简体中文。
